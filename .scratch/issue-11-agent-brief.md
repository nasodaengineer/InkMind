## Agent Brief

**Category:** bug
**Summary:** `inkmind write` 首次写作时新建的 `PipelineState` 未回写 `current_chapter_index` / `total_chapters`，导致同一 novel 第二次 write 的 `chapter_index` 重复为 1。

**Current behavior:**
`write` 命令只在「pipeline 已存在」的分支里回写 `current_chapter_index = next_index` 与 `total_chapters = max(total_chapters, next_index)`；新建 `PipelineState` 的分支保持 `current_chapter_index=None`、`total_chapters=0`。因此第二次 write 计算下一章序号时 `(None or 0) + 1 = 1`，与首章重复；第三次才得到 2。

**复现证据（2026-07-17 triage 实测，已通过）：**
对同一 novel 连续执行三次 `inkmind write`，实际输出 `chapter_index` = **1, 1, 2**，与本 issue 描述一致。

**Desired behavior:**
无论 `PipelineState` 是新建还是已存在，`write` 完成后都应一致地记录本次章节序号：`current_chapter_index` 等于本次写入章节的 index，`total_chapters` 不小于该 index。可参考 `CollaborationPipeline`（`next` 命令路径）中既有的模式——新建分支只负责构造对象，序号回写在分支外无条件执行。对同一 novel 连续执行 N 次 write，章节序号应依次为 1..N。

**Key interfaces:**
- `PipelineState`（`inkmind.models.agent` 领域模型）— 字段语义不变，修的是 `write` 命令的回写时机，不改模型定义
- `WriteCommand._run` — 新建/已存在两个分支在 `save` 前对 `current_chapter_index`、`total_chapters` 的赋值需统一（新建分支可用与 else 分支相同的赋值，或将赋值移出分支）
- 回归测试 — `tests/test_cli.py` 的 `TestWrite` 增加「同一 novel 连续 write」的序号断言（可复用现有 `_cli` / `_extract_novel_id` 辅助函数与 `--json` 输出）

**Acceptance criteria:**
- [ ] 首次 write 后 `PipelineState.current_chapter_index == 1` 且 `total_chapters == 1`
- [ ] 同一 novel 连续 write 三章，序号依次为 1、2、3
- [ ] 不同 novel 各自的首次 write 序号仍为 1（既有 `test_write_uuid_no_collision` 行为不回归）
- [ ] `tests/test_cli.py` 新增回归测试覆盖「连续两次 write 序号为 1、2」，且全套测试通过

**Out of scope:**
- `next` 命令 / `CollaborationPipeline` 的序号逻辑（经核查已正确，不在本工单改动）
- T1 幂等 digest 行为（不同内容不拦截属设计如此）
- `review` / `next` 取「最新章」的语义（本 issue 仅提及影响面，不要求改动）
